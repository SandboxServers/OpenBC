# Getting Started with OpenBC Modding

OpenBC is designed from the ground up for modding. Every game behavior -- combat rules, scoring, power simulation, repair -- runs through the same extension system that mods use. The stock game is the reference mod.

## Three Ways to Mod

| Approach | Skill Level | What You Can Do |
|----------|-------------|-----------------|
| **TOML config** | Beginner | Define game modes, missions, change settings, bind events |
| **Lua scripts** | Intermediate | Custom scoring, spawn rules, dynamic game logic, AI tweaks |
| **C DLLs** | Advanced | Replace or extend any game system, full engine API access |

You can mix all three. A typical mod uses TOML for configuration, Lua for custom rules, and optionally a C DLL for performance-critical systems.

## Your First Mod: Custom Game Mode (TOML Only)

Create a file `mods/my-mod/modes/instagib.toml`:

```toml
[mode]
name = "Instagib FFA"
description = "One-hit kills, no shields"

[mode.overrides]
frag_limit = 20
time_limit_minutes = 5
collision_damage = true
```

Tell the server to use it in `server.toml`:

```toml
[game]
mode_file = "mods/my-mod/modes/instagib.toml"
```

That's it. No code needed.

## Adding Lua Logic: Bonus Scoring

Create `mods/my-mod/scoring.lua`:

```lua
-- Award bonus points for self-destruct kills
obc.on("ship_killed", function(ctx)
    if ctx.method == "self_destruct" then
        obc.score_add(ctx.killer_slot, 0, 0, 500)
        obc.log("Bonus! Self-destruct kill by slot " .. ctx.killer_slot)
    end
end, 150)  -- Priority 150: runs after stock scoring (50)
```

Register it in `server.toml`:

```toml
[[modules]]
name = "bonus_scoring"
lua = "mods/my-mod/scoring.lua"
```

## Replacing a Stock System: C DLL

If you need to completely replace how combat works:

1. Write a C module (see [DLL Module Guide](dll-module-guide.md))
2. Build it as a DLL
3. Replace the stock module in `server.toml`:

```toml
[[modules]]
name = "combat"
dll = "mods/my-mod/combat.dll"   # Replaces stock combat module
[modules.config]
instagib = true
```

## Mod Directory Structure

```
mods/
  my-mod/
    manifest.toml       # Mod metadata (name, version, author)
    modes/              # TOML game mode definitions
      instagib.toml
    missions/           # TOML mission definitions
    scripts/            # Lua scripts
      scoring.lua
    modules/            # C DLLs (optional)
      combat.dll
    data/               # JSON data packs (optional, machine-generated)
      ships.json
```

### Mod Manifest

Every mod needs a `manifest.toml`:

```toml
[mod]
name = "Instagib Mode"
version = "1.0.0"
author = "YourName"
description = "One-hit kills with bonus self-destruct scoring"
compatible_api = 1
```

## Loading Mods

In `server.toml`, mods are loaded by referencing their files directly in module entries:

```toml
# Stock modules load first (from modules/ directory)
[[modules]]
name = "protocol"
dll = "modules/protocol.dll"

[[modules]]
name = "combat"
dll = "modules/combat.dll"

# Mod modules load after stock
[[modules]]
name = "bonus_scoring"
lua = "mods/my-mod/scoring.lua"
```

Module order in `server.toml` determines load order. Stock modules typically come first.

## Event System Basics

Everything in OpenBC communicates through events. When a ship is killed, the engine fires a `ship_killed` event. Any module -- stock or mod -- can subscribe to that event.

Key concepts:
- **Priority** determines execution order (lower = runs first; stock = 50)
- **Cancellation** lets a handler prevent lower-priority handlers from running
- **Relay suppression** lets a handler prevent network relay of an event

See [Event System](../architecture/event-system.md) for the full event catalog.

## Ship Data Packs

If your mod adds new ships, you provide a JSON data pack (machine-generated by the ship scraper tool):

```toml
[data]
registry = "data/vanilla-1.1/"
mod_packs = ["mods/my-mod/data/"]
```

Mod data overlays the base data. If both define species ID 5, the mod's version wins.

## Testing Your Mod

1. Start the server with your mod loaded
2. Connect a stock BC 1.1 client
3. Check the server log for errors
4. Verify your handlers fire by watching log output
